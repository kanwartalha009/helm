# Decisions

Ratified decisions that deviate from, or extend, the signed spec (`docs/00–13`).
The spec stays **intact** as the contracted vision and the architecture we build
toward; this log records where the running build has knowingly diverged, and why.

Rule: a new deviation gets a row here the same day it is ratified. The spec is
never silently overwritten — if reality and a spec section disagree, this log is
the tiebreaker for "what is true today," and the spec remains the tiebreaker for
"what we are building toward."

| D-026 | 2026-07-12 | **Sessions by traffic type: stored per landing ENTITY, paged not capped** (Bosco inputs §B, probe verdict B1). Deviations from spec §B.2: (a) `session_traffic_daily` is keyed on **(brand, date, entity_type, entity_key, traffic_type)** — the landing path is resolved to a product handle / collection handle / `'store-wide'` at SYNC time — **not** on the raw `landing_path` the spec specifies; (b) the sync **pages with LIMIT+OFFSET until a short page**, rather than the spec's "top-N landing paths per day (default 200) + truncation note"; (c) every day is **reconciled** against a second ShopifyQL call (`GROUP BY traffic_type`) and stored `is_complete=false` if the paged sum ≠ the store total; a window with any unreconciled day renders **"—", never a partial sum**; (d) backfill runs **day by day**, not month-chunked like `shopify:backfill-funnel`; (e) the day is written as an **atomic delete+insert in a transaction**, not an upsert; (f) **five** traffic types are rendered (paid/direct/organic/unknown/**unattributed**) — corrected 2026-07-12 after Bosco's screenshot: a 30-day probe showed only four and I wrongly ruled `unattributed` out, but a full-year probe returns it (7 sessions in 6,931,484 = 0.0001%). The first cut dropped it AFTER `pagedTotal` was summed, so reconciliation still passed while stored rows summed short — silent loss. A sixth unknown type now fails the day; (g) `LandingPathMapper` becomes the single owner of the product-handle regex, with `AdProductFetcher::productHandle()` delegating to it | feature-specs/bosco-inputs-2026-07-12.md §B; probe evidence in feature-specs/growth-os-execution-tracker.md | (a) MEASURED: one brand-day yields 2,501–5,000 distinct (path × traffic_type) rows, and the row at OFFSET 2500 was `/checkouts/cn/<unique-token>` with 1 session — every checkout mints a unique URL, so raw `landing_path` has **unbounded cardinality**. The spec's key would write ~3.5k rows/brand/day (≈100M rows/year across ~80 live brands) on a live MySQL box, almost all single-session junk. Resolving at sync time folds the whole tail into 4 rows/day and collapses locale variants (`/es/products/jay` = `jay`). (b) MEASURED: a 300-row cap captured only **74.3%** of product-page sessions, and 100% of the loss was in the tail — i.e. exactly the low-traffic products an inventory tool exists to surface. (c) a short row set that *looks* complete is the failure mode that matters: it silently under-reports a product and then the table is SORTED by that number. (e) `pulled_at` is second-precision, so a "delete rows this run didn't touch" comparison is racy inside a single second. (f) we do not manufacture a bucket Shopify has never returned; the four sum EXACTLY to the store total (20,590+9,412+3,114+2,109 = 35,225). (g) two copies of the regex could drift and split one product's numbers across two rows | Ratified 2026-07-12 |

## When to split this into files

One table is enough today. When a single decision needs real depth (context,
options weighed, consequences), give it its own `D-0XX-slug.md` and leave the row
here as the index. Keep the table as the at-a-glance map either way.
